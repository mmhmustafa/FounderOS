### EPIC-002 / PR-047 - Enterprise Policy Engine (SENTINEL)

- **Atlas's first application built on CORTEX**, and the proof the framework works. PR-046 designed the reasoning layer; this PR implements the **minimal kernel** (`founderos_atlas/reasoning/`) and builds a **greenfield Policy engine** (`founderos_atlas/policy/`) entirely on top of it — the Phase-5 acceptance gate. It passes: the new module reasons through the shared engine with **zero scoring code of its own**.
- **The kernel changes no number Atlas already computes.** A characterisation test proves the new confidence calculus reproduces `root_cause.calculate` **byte-for-byte** — the paper proof from REASONING_ENGINE §6, now executed. The kernel is `Confidence`/`ConfidenceFactor` + `clamp(base + Σ factors)` + one `band()` (bounds and bands promoted from `root_cause/`), the `Evidence`/`EvidenceGap` types, the `ReasoningResult`/`ReasoningQuestion` schema, the source-agnostic `EvidenceProvider` port, and `Rule`/`RuleRegistry`/`ReasoningEngine`.
- **Policies are data, not code.** A `Policy` declares what compliant looks like (`check`, `evidence_required`, `severity`, `expected_state`, `recommendation`, `remediation`, …); five fixed operators (`any_present`, `all_present`, `none_present`, `conditional_present`, `interfaces_shutdown`) are the whole interpreter. Compliance is one **policy pack**; Security, CIS, STIG, PCI, and customer packs install as definitions with no engine change.
- **Starter Pack — 12 policies** across configuration, routing, security, identity, management, and operational categories (SSH, NTP, Logging, AAA, SNMP, unused-interface shutdown, loopback, hostname, domain, password-encryption, BGP/OSPF router-id).
- **Every verdict is evidence-based, confidence-scored, and explained.** A failed policy carries the evidence used, the masked configuration lines, the operator that fired, a remediation with rationale, and the alternative it rejected. A device whose configuration Atlas has not collected is reported **Unknown** (band unknown), never guessed — and Unknown results are excluded from the compliance-score denominator.
- **New `/policy` page** (GUI, 127.0.0.1): compliance score + passed/warnings/failed/unknown/devices tiles, each result expandable to its full reasoning. **Evaluation runs against the masked config**, so no secret enters the reasoning layer or a rendered page; because the policies match structural directives, masking changes no verdict.
- **Validated on the live 9-device FRR lab:** 58% score over 108 evaluations, dispositions correct and honest (SSH passes from the observed transport; AAA/NTP/SNMP/domain/password-encryption fail because FRR does not express them). "Modify config → verdict flips" is pinned by test.
- **Tests:** new `tests/test_enterprise_policy.py` (27). **Full suite 1351 passed, 1 skipped, 105 subtests — no regression.**
- **No existing engine was migrated** (that is roadmap Phases 6–8, each behaviour-preserving and deleting its predecessor). Not committed, per mandate.

### EPIC-002 / PR-046 - Enterprise Reasoning Framework (CORTEX) — architecture only

- **New reference document: [`docs/architecture/REASONING_ENGINE.md`](docs/architecture/REASONING_ENGINE.md).** An architecture proposal for the reasoning layer every future module will share. **No code was written** — this PR is design, grounded in a review of the existing engines.
- **Central finding:** the shape of Atlas's reasoning is already universal; only the implementation is duplicated. All **eight** scoring models (root_cause, prediction, correlation, federation, knowledge, health, risk, network-identity) compute `clamp(base + Σ named factors)`, and all four result types express the same five concepts under different names. CORTEX is therefore an **extraction, not a rewrite** — it changes no conclusion Atlas reaches today.
- **Three findings that block building on the current structure:** (A) the shared confidence primitive lives inside `root_cause/`, a leaf analysis module five siblings import from — kernel-by-squatting; (B) Mission's reasoning lives in `web/mission.py`, i.e. inside the presentation layer, which is exactly where this session's two real bugs hid; (C) the proven `EnterpriseKnowledge` contract reached only two of seven consumers — publishing a contract is not the same as retiring the alternatives.
- Documents the reasoning **result schema** (conclusion · confidence score+band · evidence used/missing/conflicting · reasoning path · **alternatives rejected** · recommendations · severity · provenance · `as_of`), the **scoring calculus** with a standard factor vocabulary, the source-agnostic **EvidenceProvider** port (Syslog/SNMP/NetFlow/telemetry plug in with no engine change), the **rule framework**, **module contracts**, and the **AI boundary** (the LLM receives a `ReasoningResult`, never raw evidence — it explains, it never decides).
- **Roadmap gate:** build the greenfield **Compliance** module on CORTEX *first*, with zero new scoring code. If that fails, revise the framework before touching any working engine. Every later migration must delete its predecessor, or R2 (partial adoption) repeats.
- Includes 8 diagrams, 11 deliverables, 8 risks, 7 open questions, and an evidence index citing every claim to `file:line`.

### EPIC-002 / PR-045 - Enterprise Memory Foundation (MEMORY)

- **Atlas gains durable memory.** A new `enterprise_memory/` package persists every discovery session, the exact bytes every command returned, and every configuration snapshot — immutable, content-addressed, deduplicated. This is the foundation future modules (Mission, Advisor, Prediction, Investigation, Compliance, Incident, AI) will reason over without rediscovering devices. Nothing consumes it yet, by design.
- **Discovery sessions are first-class** (Part 1): id, network, profile, started/completed/duration, mode, seeds, device/authenticated/configuration/evidence counts, status — begun `running` before discovery and finalized `completed` after, with counts derived from the store.
- **Raw evidence is preserved verbatim** (Part 2): every command's output stored gzip-compressed in a content-addressed blob, stamped with a `parser_version` so a future parser can reprocess history without reconnecting to a device. Collection is separated from interpretation. Empty/unavailable output is still recorded (that a command produced nothing is itself evidence).
- **Configuration snapshots without double storage** (Parts 3–4): a snapshot is an index over the `show running-config` evidence blob, addressed by the same hash — so a configuration is never stored twice. Identical content across devices or re-runs writes no second copy; a `BlobObservation` records first/last-seen, count, and referencing sessions. Measured on the real 9-device lab: 63 evidence records → 51 unique blobs, 12 deduplicated.
- **Captured during discovery, no second session** (Part 9): the `raw_outputs` the driver already collects (and previously discarded) are caught by an optional evidence sink threaded through `discover_multihop → live → pipeline`, default off so every existing test is unchanged. Memory is a guarded side effect — it can never break discovery.
- **Stable retrieval API** (Part 7) — `EnterpriseMemory`: sessions, device memory, configuration history, raw evidence, with `view_*` (masked through the existing secret masker) vs `download_*` (raw, for the local operator). Deliberately UI-free.
- **Simple Memory UI** (Part 8): `/memory` Discovery History, session details, per-device memory with raw-evidence view (masked) and download (raw). Intentionally plain.
- Tests: **29** new (`tests/test_enterprise_memory.py`), masking mutation-verified. Full suite **1313 passed, 1 skipped, 105 subtests** (+29, no regression).

### EPIC-002 / PR-044B - Universal Web Management Access (PORTAL)

- **From seeing a managed device to opening its web UI, in one click.** A new `management/` package and an extension to the same universal device-action macro let an engineer open a device's verified HTTPS (or, clearly-marked-insecure, HTTP) management interface from anywhere the device appears. Reuses PR-044A's canonical-device resolution and action component; no per-page device-action logic.
- **HTTPS is always preferred.** When a verified HTTPS endpoint exists it is the primary action; HTTP appears only when it is the only way in, labelled *Open HTTP — Insecure* and gated behind a per-device confirmation. Custom ports (443/8443/9443/80/8080) are configurable and preserved exactly in the URL.
- **A listening port is only a candidate.** `management/verify.py` marks a service *verified* only after it answers as a web interface — a TLS handshake or open port with nothing behind it is a candidate, never verified. Detection is bounded (short timeouts) and concurrent, designed to ride after discovery with no material delay.
- **Canonical identity is the security root, reused from the console.** A web URL is only ever built on the address Atlas authenticated to over SSH. A router ID, BGP peer, next hop, unverified loopback, or unresolved peer is never used — and a stored service naming any other address is rejected at resolve time.
- **Honest TLS.** Two handshakes (`management/certs.py`): a permissive read of what the device presented, and a verifying check against the trust store. Atlas collects subject/issuer/SANs/validity/fingerprint/trust and flags self-signed, expired, hostname-mismatch, untrusted-CA and certificate-changed. It never suppresses the browser's own TLS warning and never claims an unverified certificate is safe. Verified live against real self-signed, mismatched, and expired certificates.
- **No credential ever leaves Atlas.** A URL never carries one, stored credentials are never auto-submitted into a device UI, nothing is proxied. Every state-changing POST is behind the console's same-origin gate; opening a UI is audited (user, device, URL, outcome) — never a password, cookie, or form field.
- **Operator-defined endpoints** let an engineer state a management URL Atlas could not verify, stored clearly distinct from auto-verified services and surviving re-probes.
- New **Management** page; web actions also on Topology, Device details, Configuration, and — as a suggestion only — Advisor. Tests: **33** new (`tests/test_management.py`), security assertions mutation-verified. Full suite **1284 passed, 1 skipped, 105 subtests** (+33, no regression).

### EPIC-002 / PR-044A - Universal Device SSH Access (CONSOLE)

- **From observing a device to interacting with its CLI, in one click.** A new `console/` package and a universal device action let an engineer open an interactive SSH session to any managed device from Topology, the enterprise inventory, Device details, Configuration, Paths (per hop), and — as a suggestion only — Advisor. One Jinja macro (`_device_actions.html`) and one request-scoped resolver back every surface; no page implements its own SSH logic.
- **Canonical identity is the security root** (`console/resolve.py`). A session may be opened ONLY to an address Atlas authenticated to during discovery. A router ID, BGP peer, next hop, loopback, interface address, or unresolved peer is a protocol fact, not a way in, and is never offered. Verified on the live lab: `core1` resolves to its management endpoint 172.20.20.10, never its router ID 10.4.255.11.
- **The secret never leaves the server.** The browser holds an xterm.js terminal and exchanges bytes over a WebSocket; the password is read from the credential store at connect time, handed to paramiko, and the session drops its own reference immediately. It appears in no HTML, URL, log, browser storage, or WebSocket frame.
- **WebSocket access control** (`console/security.py`). Because WebSockets bypass the same-origin policy, an unauthenticated local SSH endpoint would be reachable by any website the operator visits. Enforced: a strict Origin/Host allowlist (cross-origin and missing Origins refused; DNS-rebinding defeated) and a single-use, device-bound, ~30s console token minted by a same-origin POST and redeemed once. An operator-identity hook (`require_operator`) is the seam a future login fills; it is honest today that trust rests on machine access, not a verified user.
- **SSH host-key verification** (`console/hostkeys.py`) — Atlas had none anywhere before. Trust-on-first-use with explicit consent: a new key shows its `SHA256:…` fingerprint and is never auto-accepted; a changed key is BLOCKED with both fingerprints shown ("rebuilt, replaced, or intercepted"); acceptance trusts the key the operator was shown. Verified live against `core1`.
- **Interactive, engineer-controlled, multi-platform.** The console carries bytes and does not parse platforms (FRR vtysh, Cisco IOS/IOS-XE, Linux shell all exercised; FRR driven live to `configure terminal`). Atlas's read-only posture governs what Atlas does autonomously, not what the operator types. Session limits (idle timeout, max duration, max concurrent), explicit disconnect, and cleanup after browser closure are enforced; a connection audit records that a session happened — operator, device, endpoint, credential reference, times — never commands or output.
- **A real bug the live run caught** that green functions had not: clicking Connect twice stacked a second SSH session beside the first (visible as an unpaired event in the audit log). Fixed and pinned by a regression test.
- **Clicking a managed node in the topology graph** offers Open SSH Console / Device Details / Configuration History / Investigate / Predict. Unresolved peers never get a session (the detail panel returns before the actions), and the standalone artifact renders none of them when Atlas is not around it. Graph-node actions appear after the next discovery, since `atlas_topology.html` is a discovery-time artifact; the device inventory on the same page offers the console today.
- Fixed: **Copy SSH Command was dead on 8 of 9 pages** — the handler lived in the terminal-only script while nine templates render the button. Moved to `atlas-device-actions.js`, loaded on every page. And a **failed copy no longer reports success**: one callback served both the resolve and reject paths, so a refused clipboard write still said "Copied"; it now falls back to a textarea copy and says "Press Ctrl+C" if that fails too.
- Tests: **75** new (`tests/test_console.py`), security assertions mutation-verified. Full suite **1251 passed, 1 skipped, 105 subtests** (+75, no regression). New optional deps under `[web]`: `flask-sock`, `paramiko`; xterm.js vendored (MIT, offline — no CDN).

### EPIC-002 / PR-044 - Configuration Intelligence (MEMORY)

- **Atlas gains memory.** Discovery answers "what exists right now?"; Configuration Memory answers "what existed yesterday, what changed, and when?". Every authenticated discovery that collects configuration now produces a durable, versioned, content-addressed historical observation — Configuration Memory becomes a first-class capability alongside the Enterprise Knowledge Graph. Built on the existing collector, section-diff, and secret-masking rather than duplicating them.
- **Content-addressed storage with duplicate suppression** (Parts 2, 3): new `config_memory/store.py`. A configuration's identity IS the SHA-256 of its text (`atlas-config:<sha>`), so identical configuration is stored **once**: a device reporting unchanged configuration records another *observation* (`first_seen` / `last_seen` / `observation_count`) rather than a duplicate copy or a fake version. Two devices with identical configuration share one blob; a device that reverts re-uses the earlier blob while still recording a new version (a revert is chronologically a real change). Measured on a 3-discovery run: **6 observations → 3 unique configurations stored, 3 deduplicated**.
- **Snapshot metadata** (Part 2): timestamp, discovery profile, network, device, platform, software version, hostname, configuration hash, snapshot id, discovery session — **provenance only, never configuration content**. Text lives solely in the local blob store (sensitive, same posture as the existing `configs/` artifacts).
- **Version history + timeline** (Parts 4, 7): per-canonical-device `v1…vN` with date/time/discovery/hash/observation counts, plus an enterprise change timeline grouped by day. A first sighting is reported as a **baseline, not a change** — Atlas never claims something changed when it has nothing to compare against; a missing blob degrades honestly ("semantic detail unknown") rather than guessing.
- **Text difference** (Part 5): deterministic side-by-side added/removed/equal via `difflib`, with optional context trimming. **Every rendered line is masked** through the existing `config_intelligence.mask_line`, so a diff view can never surface a password, key, or community string.
- **Semantic difference** (Part 6): configuration differences become **structured events** — BGP neighbor added/removed/modified, OSPF area added/removed, interface shutdown/enabled/IP/description changed, VLAN added/removed, VRF added/removed, ACL changed, HSRP priority changed, route-map/static-route/NTP/logging changes, SNMP/AAA toggles — each with severity and category. Semantics are derived from **normalized facts, not diff text**, so meaning survives formatting noise (a device re-emitting byte counts and save timestamps produces **zero** semantic events).
- **Structured configuration extraction** (Part 8): new `config_memory/extract.py` normalizes configuration into secret-free facts — hostname, router ID, AS number, BGP neighbors, OSPF areas, VLANs, VRFs, ACLs, interfaces (description/IP/shutdown/VRF), HSRP groups, NTP, SNMP, logging, AAA, static routes, route maps. Services record **that they exist, never their credential**. One rule set covers Cisco IOS/IOS-XE and FRRouting; absent constructs stay absent (never defaulted into existence).
- **Policy-driven collection integrated into discovery** (Parts 1, 9): `always | scheduled | manual | discovery-only | disabled`, each decision a pure function of (policy, context) that **states its reason**; backward compatible with the legacy `collect_configuration` boolean (True→always, False→disabled). Collection rides the existing discovery pipeline — no duplicate discovery passes — and every collected configuration is recorded into the scope's memory, respecting scope isolation.
- **Configuration UX** (Part 11): a new **Configuration** section — remembered devices, an enterprise change timeline grouped by day with severity, per-device version history, structured Configuration Knowledge, semantic changes, masked side-by-side text comparison, and raw export for the local operator.
- **Foundation only** (Part 10): Mission/Advisor/Prediction/Investigation/compliance/rollback consumption is deliberately NOT implemented — this PR builds the memory they will later read.
- Tests: `tests/test_config_memory.py` (**38 tests** — hash stability & content addressing, snapshot metadata completeness + secret-freedom, duplicate suppression, new versions, revert-reuses-blob, cross-device blob sharing, dedup statistics, exact round-trip, version history, text diff + masking + context trimming, IOS & FRR extraction, secret-free extraction, determinism, semantic events both directions, severity ordering, formatting-noise immunity, timeline baseline/enterprise/grouping/honest-degradation, all five collection policies, and discovery integration incl. unchanged-rediscovery and disabled-collection). Full suite: **1152 passed, 1 skipped, 105 subtests** — no regression.

### EPIC-002 / PR-043.9 - Network Identity & Scope (IDENTITY)

- **Atlas now distinguishes Enterprise → Network → Discovery Profile** — a product-consistency PR, no new discovery architecture. A **Network** is a logical enterprise environment (it owns topology, history, changes, predictions, health, incidents); a **Discovery Profile** is only an *observation point* (credentials, discovery method, seed/CIDR, schedule, observation metadata). The same estate scanned twice is one network, not two.
- **Network identity from evidence, never names** (`enterprise/network_identity.py`): a `NetworkFingerprint` is the normalized technical evidence of a network — serial numbers, router IDs (device metadata + the address-ownership index), loopback addresses, management addresses, seeds, interface addresses, hostnames. `compare_fingerprints` scores similarity (0–100) by weighted evidence overlap with reasons; a shared chassis serial floors the score (globally-unique hardware). The profile name is never consulted — "Delhi lab" and "Delhi lab1" that share evidence are one network; two profiles both named "core" that share none are two.
- **Duplicate network detection, never auto-merge** (Part 3): `detect_duplicate_networks` surfaces likely-duplicate observation-point pairs above a 70% threshold with the similarity score and the evidence behind it, offering **Keep Separate / Merge Later** only. `resolve_networks` clusters observation points into Networks by evidence (deterministic union-find over the similarity graph). Measured: two independent discoveries of the Delhi lab → **100% similar, 2 Networks / 3 Profiles / 1 duplicate candidate** (was 3 separate "networks").
- **Scope is authoritative everywhere** (Parts 1, 7): a new `scoped_world(scope_id)` gives each consumer the graph for the *selected* scope — at a network scope every module (Mission, Advisor, Topology, Prediction, Investigation, History, Changes) reads only that network's graph; at Enterprise they read the federated graph. **Advisor** was answering enterprise-wide while Mission showed one network — it now consumes the scoped graph (and the profile's own richer snapshot), so a scoped answer can never disagree with the scoped Mission.
- **Discovery Profile management** (Part 4): profiles now support **Run / Edit / Duplicate / Archive / Delete**. `duplicate_profile` clones the observation config under a new identity and credential (it will be flagged as a duplicate-network candidate once it discovers); `archive_profile` hides a profile from active discovery and enterprise aggregation without deleting it; **Delete removes only the observation profile** — the Network and its Enterprise Knowledge survive if another profile observes it. Backward-compatible `archived` field on `DiscoveryProfile`.
- **Mission reports operational reality** (Part 5): unused CIDR addresses are discovery statistics, **never topology changes**. The change detector's discovery-failure path now reads `discovery_statistics` and reports only genuine authentication failures (one aggregate change), not one "Discovery failed for X" per empty address. ### EPIC-002 / PR-043.3 - Discovery Execution Engine (TURBO)

- **Instrument-first, then optimize on measured evidence.** Instrumentation of the existing engine confirmed discovery is I/O-bound: wall-clock time ≈ Σ(per-candidate TCP connect + SSH handshake + auth + per-command round-trips), processed strictly sequentially, so every dead address's timeout is paid back-to-back. ### EPIC-002 / PR-037 - Atlas Path Intelligence (FLOW)

- Atlas answers the question every engineer starts with: **"why can't A reach B?"** New `founderos_atlas/path_intelligence/` domain — the first vertical slice of end-to-end connectivity investigation. Given a source and a destination device, `investigate_path()` constructs the known path from discovered topology (CDP/LLDP edges in the current snapshot), validates every hop against collected evidence, **stops at the first deterministic failure, explains WHY with cited evidence**, and recommends the next action. Deterministic only: no packet simulation, no traceroute, no AI, no guessing.
- **Hop validation pipeline**: device exists → management reachability (the last discovery's failed hosts are direct evidence) → ingress/egress interfaces exist in the collected inventory → operational state (up / down / administratively down). Each `HopResult` carries status (Pass/Warning/Failed/Unknown), ingress/egress interfaces, link state, management state, per-hop banded confidence, cited evidence, and missing evidence. Hops after a failure are marked **"not evaluated" — never assumed healthy or broken**.
- **Deterministic failure vocabulary** (protocol failures are never invented): interface-down, administrative-shutdown, missing-topology-edge, device-unreachable, discovery-incomplete, unknown-path, unknown-device, unknown-destination, ambiguous-topology. Each failure type carries its own evidence-based recommendations (physical-layer checks for a down link, "find out why it was shut" for an administrative shutdown, power/credentials/discovery for an unreachable device, fresh discovery with depth/seeds/credentials for incomplete evidence). A captured running configuration for the failing device is cited when available.
- **Honesty guarantees**: equal-cost paths are reported as **AMBIGUOUS with every candidate enumerated** — Atlas never guesses which path traffic uses; unknown devices and unreachable destinations are explained, not fabricated; devices known only from neighbor announcements are flagged as unvalidated warnings; missing interface records lower confidence and appear as unknowns; stale snapshots cost a documented confidence penalty; overall confidence is the minimum over evaluated hops, capped at 95%.
- **Investigation story**: every result narrates the investigation as numbered steps (locate endpoints → construct path → validate each hop → conclusion), each step carrying its status and evidence — the timeline a senior engineer would have written down.
- **Service + history** (`investigate_path_for_scope()`): runs against a profile scope's real artifacts (snapshot, history freshness, last run's failed hosts, captured configurations), writes `path_investigation_report.json`/`.md`, and appends the **complete result** to `path_investigations.json` (newest first, capped at 50) — timestamp, profile, source, destination, evidence, result, confidence — so any past investigation can be replayed exactly. All new artifacts are scope-local and gitignored.
- **GUI**: new **Paths** page — source/destination device dropdowns from the active network's latest snapshot, an **Investigate Path** button, and an expandable timeline (green Pass / yellow Warning / red Failure / grey Unknown) with per-hop evidence, confidence, link and management state, plus "Where communication stops", recommended next actions, what Atlas cannot see, and the recent-investigations table. ### EPIC-002 / PR-035 - Evidence-Based Root Cause Analysis Engine

- Atlas now explains **why** — deterministically. New `founderos_atlas/root_cause/` package: an evidence-based reasoning engine (no AI, no LLM, no guessing; identical evidence yields byte-identical explanations, proven by test).
- **Evidence engine**: configuration, operational, topology, discovery, and incident artifacts normalized into citable evidence items (timestamp, causal rank, affected devices/interfaces, source artifact, attributes). **Timeline engine**: deterministic ordering by timestamp then causal rank (configuration → interface → protocol → topology → incident) — intra-run ordering is causal because Atlas has run-level timestamps, not per-event device clocks; the report says so explicitly rather than inventing seconds.
- **Correlation engine + internal causal graph**: edges only along documented rules over real shared devices/interfaces/adjacency — a config change links to an interface failure on the same device (stronger when the change names the interface), interface links to protocol on the same interface, and a failure links to the removal/discovery-failure of a device that was *adjacent in the previous topology*. - **Explanation engine**: human-readable reasoning chains that follow the causal graph and cite an evidence id in every sentence, ending in the conclusion with its confidence band — fully inspectable.
- **Pipeline**: every discovery writes `root_cause_report.json`/`.md` (profile-scoped, archived in history, gitignored) with a `Root cause: …` progress line. **Historical replay**: `analyze_record()` re-analyzes any archived discovery's stored artifacts and reproduces the stored explanation byte for byte — "what happened yesterday" answered from evidence.
